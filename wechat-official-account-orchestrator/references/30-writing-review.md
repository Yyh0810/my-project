# 30 写作、审稿与合规

用于正文初稿、改写、扩写、定稿、事实核查、合规、保密、终审。

## 正文结构

科技、安全、产品类文章默认结构：

1. 开头：用具体事件、问题或场景进入。
2. 背景：只补足理解所需信息。
3. 分析：解释机制、原因、影响或取舍。
4. 运营启示：说明团队应检查、调整或决策什么。
5. 产品连接：仅在有证据和场景支撑时加入。
6. 结尾：给出行动引导或下一步。

## 写作要求

- 每段只承载一个意思。
- 少用大词，多写对象、动作和结果。
- 技术内容说明前提、边界和风险。
- 产品内容先讲场景，再讲能力。
- 不编造数据、客户、结论或外部事实。
- 不确定信息标注“待确认”或写成保守表达。

## 模型协作初稿

需要模型参与成稿时，先生成写作包，再写初稿：

```bash
python scripts/content/article_writing_pack.py --topic "AI Agent 安全运营" --type security-analysis --outline article.outline.md --out article.writing-pack.json
```

模型初稿必须遵守：

- 只能使用用户材料、写作包事实台账和已确认信息。
- 不能新增客户名、数字、漏洞编号、政策条款、市场规模或产品指标。
- 不能删除 `待核验` 标记后改写成确定事实。
- 每个小节要有结论、依据、边界和读者收益。
- 初稿完成后必须进入 `article_quality_review.py`，不能直接排版或进草稿箱。

生成初稿提示词：

```bash
python scripts/content/article_draft_prompt_builder.py article.writing-pack.json --out article.draft-prompt.md
```

表达严谨性和 AI 味单独检查：

```bash
python scripts/review/article_style_rigor_review.py article.md --out article.rigor.json
```

## 必须核验的事实

以下内容不能直接当成确定事实写入定稿。没有来源时，标注“待核验”或改成保守表达：

- 漏洞编号、漏洞数量、影响范围、利用状态。
- 政策法规、监管要求、平台规则。
- 产品版本、上线状态、性能指标、能力边界。
- 客户案例、客户名称、部署效果、效率提升。
- 第三方报告、行业排名、市场规模、调研结论。
- 日期、比例、金额、数量、百分比等具体数字。

可用 `source` 组件补充依据和事实边界，但组件不能替代真实来源。

## 内容标准

优先：

- 专业、具体、克制。
- 读者能知道该查什么、改什么、决定什么。
- 标题、摘要、开头、正文、结尾承诺一致。

避免：

- 空泛趋势词和 AI 味总结。
- 反复使用模板句式。
- 术语堆砌但不帮助行动。
- 夸大产品能力。
- 绝对化承诺：最强、唯一、彻底解决、永久解决、零风险、100%。
- 恐吓式表达：不看就亏、再不做就晚、全网沦陷、震惊。
- 未授权客户暗示：真实客户、内部资料、合同、报价、未公开数据。

## 审稿维度

| 维度 | 检查 |
| --- | --- |
| 事实 | 时间、名称、版本、编号、能力、引用是否准确 |
| 逻辑 | 标题、摘要、开头、正文、结尾是否同一承诺 |
| 风险 | 客户、密钥、内部策略、未公开数据、可滥用细节 |
| 表达 | 是否夸大、绝对化、攻击性、不适合企业号 |
| 微信体验 | 段落、层级、图片位置、摘要和封面是否适合手机 |

高风险内容先处理：密钥、客户未公开信息、可复现攻击步骤、绝对化表述、贬损竞品或机构。

更细的审稿模板见 `references/35-review-rubrics.md`。需要本地结构化质检时运行：

```bash
python scripts/review/article_quality_review.py article.md --out article.quality.json
```

审稿 JSON 固定包含：

- 标题兑现度。
- 结构完整度。
- 事实可信度。
- 专业表达。
- 风险合规。
- 产品表达。
- 读者收益。
- 发布就绪度。

总分低于 4/5 或存在阻断项时，不能进入草稿箱。先修复问题，再继续排版、素材和草稿箱链路。

## 审稿驱动改稿

审稿报告生成后，优先把问题转成改稿计划：

```bash
python scripts/review/article_revision_plan.py article.md article.quality.json --rigor-json article.rigor.json --out article.revision-plan.md
```

改稿计划应包含：

- P0 阻断项：密钥、未授权客户信息、攻击复现、占位符、待核验事实。
- P1 结构和事实问题：标题不兑现、章节缺失、来源不足。
- P2 表达问题：夸大、恐吓、营销味、产品表达过重。
- 每项修订动作和通过标准。
- 重新运行审稿的复验命令。

二次改稿默认输出新文件，不覆盖源稿。改完后重新运行 `article_quality_review.py`，通过后再进入视觉计划和排版。

## 审稿后修复方式

- 事实缺来源：补来源、改保守表达，或使用 `source` 说明边界。
- 术语解释不足：用 `definition` 补概念，不要堆术语。
- 风险提示分散：用 `warning` 收束，不要制造恐慌。
- 行动建议不足：用 `checklist` 或 `steps` 给出具体动作。
- 章节结论松散：用 `summary` 收束重点。

## 阶段输出

```text
正文/修订稿：
审稿结论：可继续 / 需修改 / 暂停发布
质量评分：
主要问题：
可直接替换的修改建议：
待确认事实：
下一阶段：
```
