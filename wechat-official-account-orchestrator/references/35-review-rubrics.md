# 35 审稿质检模板

用于科技、安全、AI Agent、产品类公众号文章的内容专业度审稿。本文提供可复用 rubric；脚本 `article_quality_review.py` 负责把其中可规则化的部分转成 JSON 检查。

## 通用评分维度

| 维度 | 通过标准 | 常见问题 |
| --- | --- | --- |
| 标题兑现度 | 标题承诺能在正文中被直接回应 | 标题很大，正文只写局部；标题含产品卖点但正文无证据 |
| 结构完整度 | 有背景、分析、判断、行动建议 | 段落堆叠、小标题不足、结尾空泛 |
| 事实可信度 | 数字、漏洞、版本、政策、案例有来源或边界 | 未标来源、写“据说/网传”、把待确认当事实 |
| 专业表达 | 克制、具体、可验证 | 最强、唯一、彻底解决、零风险、震惊式表达 |
| 风险合规 | 不泄露密钥、客户、内部策略，不给可复现攻击细节 | token、AppSecret、payload、真实客户未授权信息 |
| 产品表达 | 先场景后能力，有证据和边界 | 只堆功能词，缺少使用场景和限制条件 |
| 读者收益 | 读者知道该查什么、改什么、决定什么 | 读完只有观点，没有行动 |
| 发布就绪度 | 标题、摘要、作者、图片、占位符、危险链接都处理完 | 缺元数据、仍有待补充、本地图片未处理 |

## 安全分析稿

推荐结构：

1. 事件或风险背景。
2. 影响对象和风险边界。
3. 攻击路径或风险链路的高层解释。
4. 检查清单或处置建议。
5. 产品/能力连接，仅在有场景支撑时加入。

必须避免：

- 给出可直接复现的攻击步骤、payload、利用脚本。
- 用“全网沦陷”“不修必炸”等恐吓表达。
- 没有来源地引用漏洞数量、影响范围、厂商结论。
- 暗示客户已经受影响，除非有明确授权。

通过标准：

- 风险讲清楚，但不提供滥用路径。
- CVE、日期、影响范围、数据指标都有来源或标注待核验。
- 至少给出 3 条可执行检查或处置建议。

## 产品说明稿

推荐结构：

1. 场景问题。
2. 用户/团队当前痛点。
3. 产品能力如何介入。
4. 示例路径、能力边界或结果证据。
5. 下一步行动。

必须避免：

- “一键解决所有问题”“零成本替代人工”等绝对化承诺。
- 只写功能清单，不写场景和使用方式。
- 未经证实的客户效果、效率提升、市场排名。
- 把仍在规划中的能力写成已上线。

通过标准：

- 产品能力与场景一一对应。
- 关键能力有示例、边界或证据。
- 读者知道适合什么场景、不适合什么场景。

## 行业观察稿

推荐结构：

1. 现象或趋势。
2. 为什么值得关注。
3. 对团队、产品或安全运营的影响。
4. 反例、边界或不确定性。
5. 行动建议。

必须避免：

- 只堆趋势词：大模型、智能体、重构、颠覆。
- 将个案推成行业定论。
- 没有来源地引用报告、排名、市场规模。
- 用标题制造过高预期，正文却只做泛泛评论。

通过标准：

- 至少有一个具体证据或案例。
- 明确“观察”与“结论”的边界。
- 有对读者决策有帮助的判断标准。

## 审稿后修复组件

- 事实边界不清：使用 `source` 补充依据与说明。
- 术语解释不足：使用 `definition` 解释关键概念。
- 风险提示分散：使用 `warning` 收束阻断项或注意事项。
- 行动建议不清：使用 `checklist` 或 `steps`。
- 章节结论松散：使用 `summary`。

组件只用于修复阅读和结构问题，不用于掩盖事实缺失。事实没有依据时，优先补来源或删除，不要只加样式。
